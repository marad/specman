## AC-1 (added): Given a plan file with two verification commands that both exit zero and leave the tree clean, `specman verify <FEAT-ID>` prints a success summary and exits zero.

Approach: Add a new `verifyCommand(root, featId, opts?)` function in `src/sync.ts` that orchestrates the standalone verify flow. Reuses `readPlan` / direct file read for `--plan`, `parsePlan` (src/plan.ts:198), and `runVerification` (src/sync.ts:306). On all-pass it returns a `VerifyCommandResult` with `outcome: "passed"` and a success message; the CLI prints it and exits 0. The CLI handler in `cli.ts` parses the new `verify` subcommand and passes through to `formatVerifyResult`.

Files:
- src/sync.ts (add `verifyCommand`, `VerifyCommandResult` type, `formatVerifyResult`)
- cli.ts (add `verify` case + help text)

## AC-2 (added): Given a plan file whose second verification command exits non-zero, `specman verify` stops after the second command, prints the failure details (command, exit code, stdout, stderr), and exits non-zero.

Approach: `verifyCommand` delegates to existing `runVerification`, which already stops on the first non-zero exit (src/sync.ts:318-324). Map the failed `VerificationResult` to `outcome: "command-failed"` and reuse the same per-command formatter pattern as `formatSyncResult`'s `verification-failed` branch (cmd / exit code / stdout / stderr). CLI exits 1.

Files:
- src/sync.ts (formatting branch in `formatVerifyResult`)

Approach: Already handled by `runVerification`'s dirty-tree post-check (src/sync.ts:327-334). Map to `outcome: "dirty-tree"` in `verifyCommand`; print the offending command plus the dirty paths list, mirroring the sync formatter. CLI exits 1.

## AC-4 (added): Given no plan file exists for the given FEAT-ID, the command exits non-zero naming the missing plan path.

Approach: Before parsing, `verifyCommand` resolves the plan path (default `.specman/plans/<FEAT-ID>.md` or `--plan` override) and reads the file. On `Deno.errors.NotFound` (or any read failure when the file is missing), return `outcome: "no-plan"` with a message naming the resolved path. CLI exits 1.

Approach: After `parsePlan`, call existing `checkVerificationNonEmpty` (src/plan.ts:388). If it returns a non-null error string, return `outcome: "empty-verification"` with that message. CLI exits 1 and prints the same wording as `formatVerificationEmptyError`.

## AC-6 (added): Given `--plan ./my-plan.md`, the command reads verification commands from that path instead of the default location.

Approach: CLI argument parser accepts `--plan <path>` (relative or absolute). `verifyCommand` takes an `opts.planPath` parameter; when provided, it resolves that path against `root` (if relative) instead of building the default `.specman/plans/<FEAT-ID>.md` path. Everything downstream — parse, run, dirty check — is unchanged.

## AC-7 (added): `specman verify` does not write any snapshot, create any commit, or modify any file. It is read-only except for whatever side effects the verification commands themselves produce (which are then detected by the dirty-tree check).

Approach: By construction — `verifyCommand` only reads the plan file, parses it, and invokes `runVerification`. It never calls `writeSnapshot`, `writeSnapshotCommit`, `writePlan`, or any git mutation. A test asserts that after a passing verify run, the snapshot file does not exist (or is unchanged) and `git log` is unchanged from before the run.
