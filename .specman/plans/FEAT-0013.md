# Sync plan — FEAT-0013 Install command

Started: 2026-04-28T13:41:42.728Z
Snapshot state: new
Drift summary: 16 added (whole spec)

> **Plan note:** New command. Adds `src/install.ts` + `src/install_test.ts`, four template files under `templates/`, and `install` + `uninstall` cases in `cli.ts`. Templates are authored as `.md` files and bundled into the compiled binary via `deno compile --include templates`; at runtime they are loaded with `Deno.readTextFileSync` against a URL resolved from `import.meta.url`. The `deno.json` `compile` and `install` tasks are updated to add `--include templates`.
>
> Spec language note: the spec mentions Rust's `include_str!` as the static-templates illustration. The Deno equivalent used here is `deno compile --include` plus runtime read; the spec's `(or equivalent)` hedge covers it. Worth a wording cleanup in a follow-up editorial seal — not blocking.

## AC-1 (added): Given a TTY and no positional arguments, `specman install` presents an interactive checklist of every supported agent and installs the artifacts for each agent the user confirms.

Approach (NEW CODE):
1. In `cli.ts` `install` case, when no positional agents are given, check `Deno.stdin.isTerminal()`. If true, call `interactiveSelect(agents)` (new helper in `src/install.ts`).
2. `interactiveSelect` renders a numbered list with `[x]`/`[ ]` toggles (default: all checked), reads single-line input with `prompt`, and accepts either a number (toggles that line) or empty input (confirms). Loop until empty input. Return the selected agent identifiers.
3. The numbered-toggle UX is chosen over raw-mode arrow-key navigation for portability and zero new dependencies — fits the existing stdlib-only pattern.
4. After selection, dispatch to the same install routine used by the explicit form (AC-2).

Files: cli.ts (new `install` case), src/install.ts (new file: `interactiveSelect`, `installAgents`)

## AC-2 (added): Given one or more agent identifiers as positional arguments, `specman install <agent>...` skips the prompt entirely and installs the artifacts for exactly the named agents in the order given.

Approach (NEW CODE):
1. In `cli.ts` `install` case, parse `args.slice(1)` for positional agent identifiers (anything not starting with `--`). Flags: `--global`, `--list`. No prompting if positionals are non-empty.
2. Validate each identifier against `AGENTS` (a record in `src/install.ts` keyed by identifier). Unknown identifiers fail per AC-5 before any file is written.
3. Pass the validated list, in given order, to `installAgents(agents, {scope})`. Output ordering follows input ordering (AC-11).

Files: cli.ts, src/install.ts

## AC-3 (added): Given no TTY and no positional agent arguments, `specman install` exits non-zero with an error directing the user to pass agent names explicitly, and writes nothing.

Approach (NEW CODE): In the same install case branch as AC-1, if positionals are empty AND `Deno.stdin.isTerminal() === false`, print to stderr: `error: no agents specified and no TTY available for interactive selection.\nPass agent names explicitly, e.g. \`specman install claude-code\`.` and `Deno.exit(1)` before any filesystem write.

Files: cli.ts

## AC-4 (added): Given `--list`, `specman install --list` prints each supported agent identifier on its own line, exits zero, and writes no files.

Approach (NEW CODE): In `cli.ts` `install` case, if `--list` is in `args.slice(1)`, print `Object.keys(AGENTS)` one per line in the canonical order (`claude-code`, `opencode`, `copilot-cli`) and `Deno.exit(0)`. Short-circuits before any project-root or TTY checks. The `--list` branch is checked first; positional agents alongside `--list` are ignored (the listing is the only side effect).

Files: cli.ts, src/install.ts (export `AGENTS`)

## AC-5 (added): Given an unsupported agent identifier, `specman install <agent>` exits non-zero with an error naming the identifier and pointing at `--list`, and writes nothing for that invocation.

Approach (NEW CODE): In `cli.ts` `install` case, after parsing positionals, validate each against `AGENTS`. On the first unknown identifier, print to stderr: `error: unknown agent '${id}'. Run \`specman install --list\` to see supported agents.` and `Deno.exit(1)`. No partial install — the validation happens before any file write.

Files: cli.ts

## AC-6 (added): Given the default scope (no `--global`) and a current working directory that is not inside a specman-initialized project, `specman install` exits non-zero with an error pointing at `specman init` and `--global`, and writes nothing.

Approach (NEW CODE):
1. In `cli.ts` `install` case, after `--list` short-circuit and after parsing `--global`, branch on scope.
2. For default scope: call `findProjectRoot(Deno.cwd())` (existing in `src/root.ts`). If `null`, print to stderr: `error: no specman project found (walked up from ${Deno.cwd()}).\nRun \`specman init\` first, or use \`specman install --global\` for a per-user install.` and `Deno.exit(1)`.
3. The error message lists both remediations (init + --global) per the spec example.

Files: cli.ts, src/root.ts (existing — no changes; reuse `findProjectRoot`)

## AC-7 (added): Given the default scope and a current working directory inside an initialized project, `specman install` writes the four artifacts (one skill plus three slash commands) per selected agent, into that agent's project-scoped paths.

Approach (NEW CODE):
1. Define `AGENTS: Record<AgentId, AgentBackend>` in `src/install.ts`. Each backend exports:
   - `id: string` (e.g. `"claude-code"`)
   - `projectPaths: { skill: string, commandSpec: string, commandSync: string, commandStatus: string }` (relative paths from project root)
   - `globalPaths: { ... }` (absolute paths or paths relative to user home)
   - `wrap(template: TemplateKind, content: string): string` — applies any agent-specific frontmatter wrapping (e.g. Claude Code skills need YAML frontmatter with `name` + `description`).
2. Concrete project paths at MVP:
   - `claude-code`: `.claude/skills/specman/SKILL.md`, `.claude/commands/spec.md`, `.claude/commands/spec-sync.md`, `.claude/commands/spec-status.md`
   - `opencode`: `.opencode/agents/specman.md`, `.opencode/commands/spec.md`, `.opencode/commands/spec-sync.md`, `.opencode/commands/spec-status.md`
   - `copilot-cli`: `.github/copilot/skills/specman/SKILL.md`, `.github/copilot/commands/spec.md`, `.github/copilot/commands/spec-sync.md`, `.github/copilot/commands/spec-status.md`
3. `installAgents(agents, {scope: "project", root})` resolves each backend's path against `root`, calls `wrap(kind, template)` for each of the four templates, and writes via `Deno.writeTextFileSync` (with `mkdirSync({recursive: true})` first).
4. Each path written is appended to a `written: string[]` array surfaced to the CLI for output (AC-11).

Note on per-agent paths: claude-code's are documented and stable. The exact path conventions for opencode and copilot-cli should be verified against current upstream documentation during implementation; the table above is the implementer's starting point and may shift slightly. Adjustments are AC-7-internal — they don't affect any other AC.

Files: src/install.ts (`AGENTS`, `installAgents`, four `AgentBackend` definitions), templates/skill.md, templates/spec.md, templates/spec-sync.md, templates/spec-status.md

## AC-8 (added): Given `--global`, `specman install --global <agent>...` writes the same four artifacts per selected agent into the agent's user-home paths, regardless of whether the current working directory is inside a specman project.

Approach (NEW CODE):
1. Resolve `homeDir = Deno.env.get("HOME") ?? Deno.env.get("USERPROFILE")`. Error with a clear message if unset.
2. Concrete global paths at MVP:
   - `claude-code`: `${HOME}/.claude/skills/specman/SKILL.md`, `${HOME}/.claude/commands/{spec,spec-sync,spec-status}.md`
   - `opencode`: `${HOME}/.config/opencode/agents/specman.md`, `${HOME}/.config/opencode/commands/{spec,spec-sync,spec-status}.md`
   - `copilot-cli`: `${HOME}/.config/github-copilot/skills/specman/SKILL.md`, `${HOME}/.config/github-copilot/commands/{spec,spec-sync,spec-status}.md`
3. With `--global`, skip `findProjectRoot` entirely. `installAgents` is called with `{scope: "global"}` and resolves paths against `homeDir`.
4. Output uses absolute paths (matches the spec's example: `Wrote /home/alice/.claude/skills/specman/SKILL.md`).

Files: src/install.ts (same as AC-7)

## AC-9 (added): Given a previous install whose artifacts already exist on disk, re-running `specman install` with the same arguments overwrites every artifact without prompting and exits zero.

Approach (NEW CODE): `Deno.writeTextFileSync` already truncates and overwrites by default. No pre-check, no `--force`, no backup. Each `Wrote <path>` line is printed regardless of whether the file existed before — the user-facing output is the same on first install and re-install.

Files: src/install.ts (already covered by `installAgents` in AC-7)

## AC-10 (added): Given an installed agent, `specman uninstall <agent>` removes every artifact that `install` would have written for that agent and exits zero. Missing files are not an error.

Approach (NEW CODE):
1. Add `uninstall` case to `cli.ts` switch. Parse positional agents and `--global` similarly to install. `--list` is not relevant for uninstall.
2. `--global` controls scope the same way as install. Project scope requires `findProjectRoot`; global uses home dir.
3. `uninstallAgents(agents, {scope, root})` iterates each agent's four paths, calls `Deno.removeSync(path)` and catches `Deno.errors.NotFound` (treated as success — no error). Each removed path prints `Removed <path>`. Skipped (already-missing) paths print nothing.
4. Empty parent directories left behind (e.g. `.claude/skills/specman/`) are not pruned — the user's other tooling may use them. Pruning is a separate consideration not part of MVP.

Files: cli.ts (new `uninstall` case), src/install.ts (`uninstallAgents`)

## AC-11 (added): Given a successful install, the output names every path written, in the order written, and ends with a one-line hint about restarting the agent.

Approach (NEW CODE):
1. `installAgents` returns `{written: string[]}`. CLI prints `Wrote <path>` per entry in the order they were written (agents iterated in user-specified order; within each agent, fixed order: skill, spec, spec-sync, spec-status).
2. After all writes, CLI prints exactly: `Restart your agent or open a new session to load the new skills.`
3. The hint is unconditional on success (any agent installed). On failure (AC-15), the hint is suppressed.

Files: cli.ts, src/install.ts

## AC-12 (added): Given the supported agent matrix at MVP, `specman install --list` prints exactly: `claude-code`, `opencode`, `copilot-cli` — one per line.

Approach (NEW CODE): `AGENTS` is an ordered object literal with exactly three entries in this order: `claude-code`, `opencode`, `copilot-cli`. `--list` prints `Object.keys(AGENTS).join("\n")`. Test asserts exact output (`assertEquals(stdout, "claude-code\nopencode\ncopilot-cli\n")`).

Files: src/install.ts (`AGENTS`), src/install_test.ts

## AC-13 (added): Given any installed slash command (`/spec`, `/spec-sync`, `/spec-status`), invoking it without the skill having loaded provides the agent enough context to perform the corresponding workflow correctly. The slash command bodies are self-sufficient and do not assume the skill is in context.

Approach (CONTENT AUTHORING): Author the four template files. This is a content task — the templates themselves must satisfy the AC, not any code mechanism. Content rubric:
- `templates/skill.md`: ~150 lines. Mental model + lifecycle + `specman status` reading + new-vs-update routing principle + cross-references to the three slash commands. No detailed how-to.
- `templates/spec.md`: Self-sufficient. Includes a one-paragraph mental model recap (specs are SoT, ACs are the sync interface), then the new-vs-update routing process, the new-flow (`specman new`), the update-flow (edit in place), spec-writing principles condensed (Intent, AC quality), validation loop, and closeout decision (AC drift → /spec-sync; editorial → `specman seal`).
- `templates/spec-sync.md`: Self-sufficient. One-paragraph recap, then `specman sync FEAT-X` → fill plan (Approach + Files per AC) → **explicit human approval gate** → implement → `specman verify` → seal.
- `templates/spec-status.md`: Self-sufficient. One-paragraph recap, then `specman status` / `--diff`, interpretation, recommended next action.

Each slash command starts with a small recap so the agent has the mental model even if the skill never loaded — that's the AC's testable property. Verification is by reading the templates and confirming the recap-then-workflow structure is present in each (manual + a structural test that asserts each command template begins with a recap section).

Files: templates/skill.md, templates/spec.md, templates/spec-sync.md, templates/spec-status.md

## AC-14 (added): Given any installed skill, its trigger metadata fires on at least the terms "spec", "specman", "acceptance criteria", "drift", and "sync" appearing in user input, in whatever form the agent's matching mechanism supports.

Approach (CONTENT AUTHORING):
1. The `templates/skill.md` body itself is unwrapped content. Each `AgentBackend.wrap("skill", body)` adds the agent-specific trigger metadata. For Claude Code: a YAML frontmatter `description` field whose text contains all five trigger terms verbatim — Claude Code matches descriptions against user input. For Copilot CLI: equivalent frontmatter or marker (verify exact syntax during implementation). For Opencode agent definitions: include the trigger terms in the agent's description field.
2. Test: assert that for each agent, the wrapped skill content (as it would be written to disk) contains each of `spec`, `specman`, `acceptance criteria`, `drift`, `sync` as substrings.

Files: src/install.ts (`AgentBackend.wrap`), src/install_test.ts

## AC-15 (added): Given a partial install failure (e.g. permission denied on the third file written), the artifacts already written remain on disk; install exits non-zero naming the failed path; no rollback is attempted.

Approach (NEW CODE): `installAgents` writes one file at a time, appending each successful path to `written`. On the first `Deno.writeTextFileSync` exception, it returns `{written, error: {path, message}}` without throwing. CLI prints all `Wrote <path>` lines for entries already in `written`, then prints `error: failed to write <path>: <message>` to stderr, suppresses the restart hint, and `Deno.exit(1)`. No try/catch around the loop that would attempt cleanup — successful writes persist.

Files: src/install.ts, cli.ts

## AC-16 (added): Given install runs entirely from baked-in templates, neither network access nor any external file outside the binary and the destination paths is read or written.

Approach (NEW CODE):
1. Templates are loaded via `Deno.readTextFileSync(new URL("../templates/<name>.md", import.meta.url))`. In the compiled binary, `--include templates` (added to `deno.json` `compile` task) bundles these files; the URL resolves to the embedded asset.
2. No `fetch`, no `Deno.connect`, no external process invocation in `src/install.ts`. The Deno permission set required to run install is `--allow-read --allow-write --allow-env` (env for `HOME`); `--allow-net` is intentionally not in the compile task.
3. The compile task in `deno.json` becomes: `"compile": "deno compile --allow-read --allow-write --allow-run --allow-env --include templates --output specman cli.ts"` (and the matching `install` task gets the same flags).
4. Test: a permissions-restricted test invocation that runs `install --list` with `--deny-net` confirms no network access is attempted.

Files: src/install.ts, deno.json (update `compile` and `install` tasks), src/install_test.ts
